---
name: prototype-parity
description: Use for any frontend or UX task that must preserve 1:1 parity with `.prototype`, including routes, copy, themes, states, and component behavior.
---

Checklist:

1. Identify the affected prototype routes and components.
2. Compare `.prototype` and the implementation for:
   - route coverage
   - copy
   - layout hierarchy
   - statuses and edge states
   - theme parity
3. Record a parity checklist in the task evidence.
4. If the implementation must diverge, document:
   - why
   - which work item authorizes it
   - the exact user-facing impact

Never silently "improve" the prototype.
