---
name: adr-driven-design
description: Use when the user asks for architecture decisions, tradeoffs, or ADRs in a backend context.
---

Follow this workflow for ADR-driven backend design:

1. Read the work item and architecture docs first.
2. Start a record from `templates/adr/decision-record.md` when the decision affects architecture, rollout, or ownership.
3. Keep options, decision, and consequences explicit before implementation expands.

- Do not let high-impact decisions stay implicit.
- Keep ADRs concise and decision-complete.
