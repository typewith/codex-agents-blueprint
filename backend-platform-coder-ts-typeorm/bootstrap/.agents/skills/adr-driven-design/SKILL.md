name: adr-driven-design
description: Use when the user asks for backend architecture decisions, tradeoffs, or decision records.
---

Follow this workflow for ADR-driven backend design:

1. Read the work item and architecture docs first.
2. Start a record from `templates/adr/decision-record.md` when the decision affects module boundaries, rollout, or data shape.
3. Keep alternatives and consequences explicit before implementation begins.

- Do not let risky architecture decisions stay implicit in chat.
- Keep ADRs short and decision-oriented.
